# Developer Guide: SMM Panel Architecture

This guide is intended for developers who want to modify, extend, or maintain the SMM Panel project. It covers the core design patterns, database architecture, and workflows.

---

## 1. Authentication Flow

Authentication is handled manually via JWT (JSON Web Tokens).

- **Generation:** On `/api/auth/login` or `signup`, a token is signed with the user payload `({ userId, userName, role })` and returned to the client.
- **Storage:** The frontend stores the token and user object in `localStorage`. 
- **Verification:** API routes that require protection call `authenticate(request)` from `lib/auth.js`. This function extracts the `Bearer` token from headers and verifies it.

**To add a new protected route:**
```javascript
import { authenticate } from "@/lib/auth";

export async function GET(request) {
  const decoded = await authenticate(request);
  if (!decoded) return NextResponse.json({ message: "Unauthorized" }, { status: 401 });
  
  // For admin only:
  if (decoded.role !== "admin") return NextResponse.json({ message: "Forbidden" }, { status: 403 });
}
```

## 2. Database Schemas (Mongoose)

All database logic lives in `src/models/`. If you add new models, ensure you use the fallback export pattern for hot-reloading in Next.js:
`export default mongoose.models.ModelName || mongoose.model("ModelName", Schema);`

### Categories (`models/Category.js`)
Controls the primary dropdown menu.
- `name`: String.
- `icon`: String (URL or raw HTML `<img>` tag).
- `sortOrder`: Number (controls display order).
- `isActive`: Boolean (toggle visibility).

### Services (`models/Service.js`)
The actual logic for user orders.
- `categoryId`: Reference to the Category.
- `serviceType`: Enum (`default`, `custom_comments`, `package`). Used by frontend to alter the UI (e.g., show a textarea for custom comments).
- `price`: Number. Charged to the user (per 1,000 units).
- `actualPrice`: Number. Cost to third-party provider (for profit calculations).
- `providerId` & `externalServiceId`: For future 3rd-party API integrations.
- `description`: Stores rich HTML rules generated by React-Quill.

### Orders (`models/Order.js`)
Records of user requests.
- `totalCharge`: Calculated dynamically on backend before saving `(price / 1000) * quantity`.
- `status`: Enum (`Pending`, `Processing`, `Completed`, etc.). Admin updates this manually.
- `comments`: Text. Only populated if `serviceType === "custom_comments"`.

### Users (`models/User.js`)
- `balance`: Deducted when placing orders. 
- `totalSpent`: Aggregates the total value a user has spent in their lifetime.

## 3. Adding New Features (Workflow)

If you need to add a new feature (e.g., "Add Funds via Stripe"), follow this architectural workflow:

1. **Schema Update:** Modify `models/User.js` to add a `transactions` array or create a new `models/Transaction.js`.
2. **API Route:** Create `src/app/api/payments/route.js`. Ensure you protect the route using `authenticate()`.
3. **Frontend Component:** Create the UI component inside `src/app/(user)/add-funds/page.js`.
4. **Link Update:** Add the route to the `navLinks` array in `Navbar.js` and `MobileSidebar.js`.

## 4. Styling & CSS Architecture

The project uses Vanilla CSS Modules paired with a central CSS variables system.

- **Global Variables:** All colors, spacing, borders, and animations are defined in `src/app/globals.css`.
- **Modifying the Theme:** If you want to change the primary color, open `globals.css` and change `--primary` and `--primary-light`. The entire application will automatically adapt.
- **Component Styles:** Each page and component has its own `.module.css` file (e.g., `new-order.module.css`). Always use classes from the module (`className={styles.title}`) instead of global classes to prevent scoping conflicts.

### Reusable UI Components
Always use the components located in `src/components/ui/` for consistency:
- `<Button variant="primary|secondary|danger" loading={boolean} />`
- `<Input label="..." required />`
- `<Select options={[{value, label}]} />`

## 5. Upcoming API Integration (Third-Party SMM)

When you are ready to hook this up to a real third-party SMM API provider:
1. Modify `src/app/api/orders/route.js`.
2. After deducting the user's balance and saving the `Order` to MongoDB, make a `fetch()` call to the provider's API using the service's `externalServiceId`.
3. Capture the response and save the external provider's ID to the order's `externalOrderId` field.
4. Set up a Cron Job or Webhook route to routinely check the third-party provider and update the `status` of your local orders accordingly.
